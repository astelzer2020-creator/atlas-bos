# Logging

## Purpose

Define the architecture for Atlas's **Logging System** — centralized, structured, secure log aggregation that supports debugging, compliance auditing, security investigation, and operational analytics. Logging complements metrics (ARCH-19) and tracing with **searchable, retained evidence** of platform behavior while enforcing **PII redaction**, **tenant isolation**, and **audit log separation**.

## Scope

### In Scope

- Structured JSON logging standard
- Log levels and semantic conventions
- Correlation IDs across services
- Centralized aggregation (Loki primary; ELK for audit/search-heavy)
- PII detection and redaction
- Log retention tiers (hot/warm/cold)
- Audit log separation (immutable, compliance-grade)
- Query patterns and access controls
- Integration with alerting and SIEM

### Out of Scope

- Metrics and traces (ARCH-19)
- Application performance profiling
- Customer-facing activity feeds (product feature)
- Full SIEM rule authoring (ARCH-21 overview only)

---

## Context

Atlas runs hundreds of microservices across regions, processing billions of requests. Unstructured, inconsistent logs make incident response slow and compliance audits painful. A production logging architecture must:

- Answer "what happened to request X?" in seconds
- Prove who changed what for SOC 2 / GDPR
- Never leak PII into log stores accessible to all engineers
- Scale cost-effectively with volume

### Logging vs. Audit vs. Metrics

| Signal | Purpose | Mutability | Primary Store |
|--------|---------|------------|---------------|
| Application logs | Debug, ops | Mutable (retention expiry) | Loki |
| Audit logs | Compliance, security | Append-only, WORM | PostgreSQL + S3 Object Lock |
| Access logs | API forensics | Append-only | Loki + S3 |
| Business events | Analytics | Event bus | Kafka → warehouse |
| Metrics | SLOs, alerting | Aggregated | Prometheus |

## Detailed Design

### 1. Structured JSON Logging Standard

All services emit **one JSON object per line** (NDJSON) to stdout; collected by Fluent Bit → Loki.

#### Required Fields

```json
{
  "timestamp": "2026-06-30T14:00:00.123Z",
  "level": "info",
  "message": "Invoice created successfully",
  "service": "finance-api",
  "version": "2.4.1",
  "environment": "production",
  "region": "us-east-1",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",
  "correlation_id": "corr_99fa8123",
  "tenant_id": "org_abc",
  "user_id_hash": "sha256:7a3f...",
  "request_id": "req_8f3a2b1c"
}
```

#### Optional Context Fields

```json
{
  "http": {
    "method": "POST",
    "path": "/v1/invoices",
    "status_code": 201,
    "duration_ms": 45,
    "client_ip_hash": "sha256:..."
  },
  "entity": {
    "type": "invoice",
    "id": "inv_991"
  },
  "error": {
    "type": "ValidationError",
    "message": "Amount must be positive",
    "stack": "... (omitted in production info logs)"
  },
  "custom": {}
}
```

#### Prohibited in Logs

- Raw passwords, API keys, tokens
- Full credit card numbers, SSNs
- Unredacted email/phone (use hashes or partial)
- Full request/response bodies containing PII (use `body_size` + sampling)

### 2. Log Levels

| Level | Usage | Production Default |
|-------|-------|-------------------|
| `trace` | Extremely verbose (dev only) | Off |
| `debug` | Development diagnostics | Off (sampled 0.1% per service) |
| `info` | Normal operations, business events | On |
| `warn` | Recoverable issues, deprecations | On |
| `error` | Failures requiring attention | On |
| `fatal` | Process termination | On |

**Dynamic level adjustment:** SRE can bump service to `debug` for 15 minutes via config map (audited).

| Event Type | Level |
|------------|-------|
| HTTP 2xx/3xx | `info` (sampled 10%) |
| HTTP 4xx (client error) | `warn` |
| HTTP 5xx | `error` |
| Auth failure | `warn` (+ security audit) |
| Permission denied | `info` |
| Background job retry | `warn` |
| Background job DLQ | `error` |

### 3. Correlation IDs

End-to-end request tracking across async boundaries.

```
Client → X-Atlas-Correlation-Id: corr_99fa
              │
    API Gateway (generates if missing)
              │
    ├─► Auth Service (propagates)
    ├─► Finance API (propagates)
    └─► Kafka event header: correlation_id
              │
         Automation Worker (reads from header)
```

| ID Type | Scope | Generated By |
|---------|-------|--------------|
| `correlation_id` | User-facing request chain | API Gateway |
| `trace_id` | Distributed trace | OTel SDK |
| `request_id` | Single HTTP request | First service |
| `agent_run_id` | AI agent execution | Agent Orchestrator |
| `workflow_instance_id` | Workflow | Workflow Runtime |

**Query pattern:** `{correlation_id="corr_99fa"}` returns all logs across services in Loki.

### 4. Centralized Aggregation

#### Architecture

```
┌─────────────┐
│ App Pods    │──stdout──►┌─────────────┐
└─────────────┘           │ Fluent Bit  │ (DaemonSet)
                          └──────┬──────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
              ┌─────────┐ ┌─────────┐ ┌──────────┐
              │  Loki   │ │ Audit   │ │ S3 Cold  │
              │ (hot)   │ │ Writer  │ │ Archive  │
              └────┬────┘ └────┬────┘ └──────────┘
                   │           │
              ┌────▼────┐ ┌────▼────┐
              │ Grafana │ │ Audit DB│
              │ Explore │ │ + S3    │
              └─────────┘ └─────────┘
```

| Component | Role |
|-----------|------|
| Fluent Bit | Collect, parse JSON, enrich (K8s metadata), redact |
| Loki | Primary log store; label-based indexing |
| Grafana Explore | Query UI |
| Audit Writer | Separate pipeline for audit events |
| S3 | Cold archive with lifecycle policies |

**Why Loki over ELK for application logs:**

- Lower cost at high volume (index labels, not full text)
- Native Grafana integration
- Sufficient for JSON field queries

**ELK (OpenSearch) usage:** Full-text search for security investigations and compliance exports where Loki is insufficient.

#### Loki Label Strategy

Low-cardinality labels only:

```
{service="finance-api", environment="production", region="us-east-1", level="error"}
```

High-cardinality fields (`tenant_id`, `correlation_id`) stored as **JSON fields**, queried via LogQL:

```logql
{service="finance-api"} | json | tenant_id="org_abc" | level="error"
```

### 5. PII Redaction

#### Pipeline Stages

1. **Application-level:** Structured logging SDK redacts known PII fields
2. **Fluent Bit filter:** Regex redaction for accidental leaks
3. **Ingestion validator:** Rejects logs with PII patterns (metric: `logs_pii_rejected_total`)

#### Redaction Rules

| Field Pattern | Redaction |
|---------------|-----------|
| `email` | `j***@example.com` or hash |
| `phone` | `***-***-1234` |
| `ssn`, `tax_id` | `[REDACTED]` |
| `credit_card` | `[REDACTED]` |
| `password`, `secret`, `token` | `[REDACTED]` |
| `user_id` | `user_id_hash` in production logs |

**Break-glass:** Security team can access unredacted audit logs with dual approval; session recorded.

#### PII Detection

ML-based scanner (AWS Macie-style or open source) on S3 cold archives for compliance audits.

### 6. Log Retention Tiers

| Tier | Duration | Storage | Access |
|------|----------|---------|--------|
| Hot | 7 days | Loki SSD | All engineers |
| Warm | 90 days | Loki/S3 | SRE, Security |
| Cold | 1 year (app logs) | S3 Glacier | Security, Compliance |
| Audit | 7 years | PostgreSQL + S3 Object Lock | Compliance only |

**Per-tenant log isolation:** Enterprise option for dedicated retention and export; logs tagged `tenant_id` for filtered queries.

**Cost control:**

- Sample info-level HTTP logs at 10%
- Compress cold archives (Parquet)
- Drop `trace`-level at collector

### 7. Audit Log Separation

Audit logs are **not** application logs. They are legal/compliance records.

#### Audit Event Schema

```json
{
  "audit_id": "aud_7721",
  "timestamp": "2026-06-30T14:00:00Z",
  "tenant_id": "org_abc",
  "actor": {
    "type": "user",
    "id": "user_442",
    "ip_hash": "sha256:...",
    "user_agent_hash": "sha256:..."
  },
  "action": "invoice.delete",
  "resource": {
    "type": "invoice",
    "id": "inv_991"
  },
  "outcome": "success",
  "changes": {
    "before_hash": "sha256:...",
    "after_hash": null
  },
  "reason": "Duplicate entry",
  "correlation_id": "corr_99fa",
  "integrity_hash": "sha256:chain_prev+this"
}
```

#### Properties

| Property | Implementation |
|----------|----------------|
| Immutability | Append-only table; no UPDATE/DELETE |
| Integrity chain | Hash chain per tenant per day |
| WORM storage | S3 Object Lock Compliance mode |
| Separation | Dedicated `audit-api` service; not via stdout |
| Access | `audit:read` permission; all access logged |

#### Audited Actions (Minimum)

- Authentication events (login, logout, MFA, failed attempts)
- Authorization changes (role grants)
- Data export / download
- Entity CRUD on sensitive types (financial, HR, health)
- Admin actions (impersonation, break-glass)
- AI agent tool invocations (high-risk)
- Workflow approvals/rejections
- Configuration changes

### 8. Query Patterns

#### Operational Queries (Loki/LogQL)

```logql
# Errors in last hour for a service
{service="workflow-runtime", level="error"} | json

# Trace a specific request
{environment="production"} | json | correlation_id="corr_99fa"

# Tenant error spike
{service=~"finance-.*"} | json | tenant_id="org_abc" | level=~"error|warn"

# Slow requests
{service="api-gateway"} | json | http_duration_ms > 2000
```

#### Security Queries

```logql
# Failed auth spike
{service="auth-service", message=~".*authentication failed.*"}

# Permission denied pattern
{service=~".*"} | json | message="Permission denied" | tenant_id="org_abc"
```

#### Audit Queries (SQL)

```sql
SELECT * FROM audit_events
WHERE tenant_id = 'org_abc'
  AND action LIKE 'invoice.%'
  AND timestamp BETWEEN '2026-06-01' AND '2026-06-30'
ORDER BY timestamp DESC;
```

### 9. Access Control

| Role | Application Logs | Audit Logs |
|------|------------------|------------|
| Engineer | Own services, staging all | None |
| SRE | Production (no PII fields) | Read (approved) |
| Security | All (redacted) | Full |
| Compliance | Export only | Full |
| Tenant Admin | Own tenant (product audit UI) | Own tenant audit |

Grafana datasource permissions enforce boundaries.

### 10. Integration

| System | Integration |
|--------|-------------|
| Monitoring (ARCH-19) | Log-derived metrics; exemplar links |
| Alerting | Loki ruler → Alertmanager for log-based alerts |
| SIEM (ARCH-21) | Audit + security logs streamed to SIEM |
| Incident response | Correlation ID search in runbooks |
| GDPR | Erasure requests flag `user_id`; audit retains legal basis record |

### 11. Service Logging Requirements

Every Atlas service must:

1. Use shared `atlas-logging` SDK (language-specific)
2. Emit required fields
3. Propagate `correlation_id` and trace context
4. Register service name in logging catalog
5. Pass logging lint in CI (no `console.log` with raw user data)

### 12. Volume Estimates and Capacity

| Dimension | Estimate |
|-----------|----------|
| Log lines/sec (peak) | 500K |
| Average line size | 800 bytes |
| Daily ingest | ~35 TB (pre-compression) |
| Hot storage | ~250 TB-month |

Fluent Bit autoscaling; Loki ingester HPA on ingest rate.

## Alternatives Considered

### Alternative 1: ELK Stack for All Logs

**Rejected for app logs:** Cost prohibitive at billions of lines/month. Retained for audit/search use cases.

### Alternative 2: CloudWatch / Stackdriver Only

**Rejected:** Vendor lock-in; poor cross-service correlation at Atlas scale; expensive egress.

### Alternative 3: Audit Logs in Application Log Stream

**Rejected:** Cannot meet immutability and retention requirements; mixed access controls.

### Alternative 4: Unstructured Text Logs

**Rejected:** Unqueryable at scale; inconsistent fields break automation.

## Consequences

### Positive

- Sub-second incident investigation via correlation ID
- Compliance-ready audit trail with integrity guarantees
- PII-safe logging culture with automated enforcement
- Cost-effective hot storage via Loki
- Unified query experience in Grafana

### Negative

- SDK adoption burden across all services
- Redaction may obscure debugging (mitigated by break-glass)
- Dual systems (Loki + audit store) operational complexity
- Log volume costs require ongoing sampling tuning

### Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| PII leak | Multi-layer redaction; CI lint; Macie scans |
| Audit tampering | Hash chain + WORM + separation |
| Log volume cost explosion | Sampling; retention tiers; alerts on ingest rate |
| Query performance | Label discipline; recording rules for common queries |
